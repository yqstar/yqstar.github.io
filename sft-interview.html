什么是 SFT（Supervised Fine-Tuning）？它在 LLM 训练流程中处于什么位置？

高频必考核心概念
▶
核心回答

SFT 是在预训练（Pre-training）的 Base Model 基础上，用高质量的人工标注数据（指令-回答对）进行监督学习，让模型学会遵循指令、生成符合人类期望的回答。

LLM 训练全流程：

Pre-training             → SFT              → Reward Model    → PPO / DPO
(大规模无监督)            (监督微调)          (偏好建模)        (对齐优化)
    ↓                       ↓                    ↓                ↓
Next Token Prediction  指令→回答的交叉熵    Bradley-Terry偏好  RL / 偏好优化
Base Model              SFT Model           RM                Aligned Model

典型数据量：
Pre-training: 数万亿 Token（如 LLaMA 1T+）
SFT: 1万~100万条 高质量指令-回答对
RLHF: 10万~100万条 人类偏好比较对

SFT 的核心目标：

  • 让模型从"补全文本"转变为"回答指令"
  • 注入特定的格式、风格和安全行为
  • 为后续 RLHF 提供一个稳定的初始策略

全量微调（Full Fine-Tuning）和参数高效微调（PEFT）的核心区别是什么？什么场景用哪个？

高频必考选型决策
▶
核心回答
维度Full Fine-TuningPEFT
训练参数所有参数仅少量额外参数 / 部分参数
显存需求极大（模型 × 4 + 优化器状态）小（可训练参数 < 1%）
存储成本每个任务存一份完整模型（×N）每个任务存一个小适配器（~几MB）
训练速度慢快
灾难性遗忘高（可能丢失预训练知识）低（冻结主干参数）
最终性能理论上最优接近 Full FT（差距 < 1%）
硬件门槛需要多卡 / 高端 GPU单卡可微调 7B 模型

决策框架：

  • Full FT：你有充足 GPU、数据量大、需要极致性能、只有一个任务
  • PEFT：资源受限、需要服务多个任务、需要快速迭代、想保留预训练能力
💡 目前业界趋势：绝大多数应用场景都用 PEFT（尤其是 LoRA），Full FT 一般只在预训练机构做大规模训练时使用。

SFT 中的损失函数是什么？与预训练的损失函数有什么异同？为什么要 Mask 掉 Prompt 部分的 Loss？

高频训练细节
▶
核心回答

SFT 和 Pre-training 都使用交叉熵损失（Cross-Entropy Loss），即 Next Token Prediction：

# 统一的 Next Token Prediction Loss
L = - Σ_t log P(y_t | x, y_1, ..., y_{t-1})

Pre-training: 对所有 token 计算 Loss
SFT: 只对 Response 部分的 token 计算 Loss（Mask 掉 Prompt）

为什么 Mask 掉 Prompt 部分：

  • SFT 的目标是学会"给这个指令，应该这样回答"，而不是"学会预测指令文本"。
  • 如果对 Prompt 也计算 Loss，模型会浪费容量去学习重复/补全指令，而非学习指令-回答的映射。
  • 不 Mask 的情况下，Prompt 和 Response 的 Loss 混合会稀释关键的学习信号。
# 实现：labels 中 Prompt 部分设为 -100（PyTorch CrossEntropy 忽略 -100）
tokens = ["<|user|>", "你好", "请", "介绍", "北京", "<|assistant|>", "北京", "是", "..."]
labels = [-100,      -100, -100, -100, -100, -100,             "北京", "是", "..."]
#         ↑ 全部忽略                                        ↑ 从这里开始计算 Loss

什么是灾难性遗忘（Catastrophic Forgetting）？微调时如何缓解？

核心问题
▶
核心回答

灾难性遗忘：在微调过程中，模型为了适应新任务的数据分布，过度调整参数，导致在预训练阶段学到的通用知识和能力被覆盖或丢失。

缓解策略：

  1. PEFT（LoRA / Adapter）：冻结主干参数，只训练少量新增参数。最有效的防遗忘方法。
  2. 较小的学习率：通常 SFT LR ≈ 2e-5（预训练的 1/10 ~ 1/100），减少参数更新幅度。
  3. Data Replay / Mixing：在微调数据中混入一定比例的预训练数据（如 5-10%），让模型"复习"原有知识。
  4. Early Stopping：只训练 1-3 个 Epoch。SFT 通常不需要多 Epoch（容易过拟合）。
  5. KL 正则化：加一个 KL(π_θ || π_pretrained) 惩罚项，防止新策略偏离预训练分布太远。
  6. 渐进式解冻（Gradual Unfreezing）：先解冻顶层，逐层向下解冻。

多任务微调（Multi-Task Fine-Tuning）与单任务微调有什么不同？有什么优缺点？

策略选择
▶
核心回答
维度单任务微调多任务微调
数据一个任务的数据多个任务的数据混合
输出一个专用模型一个通用模型
单任务性能最优接近单任务（有任务干扰）
泛化能力差（过拟合单任务）好（多任务正则化效应）
维护成本高（每个任务一个模型）低（一个模型服务所有任务）
适用场景追求极致性能的单一场景通用助手、平台级服务

关键技巧：

  • Task Prompting：每个任务前加任务描述（"请翻译："、"请总结："），帮助模型区分任务。
  • Data Balancing：不同任务的数据量要平衡，防止模型偏向数据量多的任务。常用 Temperature Sampling 或手动调整采样比例。
  • Task-Specific Adapters：共享主干 + 每任务一个 LoRA Adapter（组合 PEFT 的多任务方案）。

LoRA（Low-Rank Adaptation）的核心原理是什么？为什么低秩分解能有效微调大模型？

高频必考核心算法
▶
核心回答

核心假设：模型适配过程中的权重更新 ΔW 是低秩的（Low-Rank）——即可以用两个小矩阵的乘积来近似。这个假设来自 Aghajanyan et al. (2020) 的 Intrinsic Dimensionality 研究。

# LoRA 核心公式
h = W₀·x + ΔW·x
  = W₀·x + (B·A)·x        # ΔW = B·A 低秩分解

其中：
W₀ ∈ ℝ^(d×k)   — 原始权重（冻结，不训练）
A  ∈ ℝ^(r×k)   — 降维矩阵（随机高斯初始化）
B  ∈ ℝ^(d×r)   — 升维矩阵（零初始化，训练开始时 ΔW=0）
r  ≪ min(d,k)  — 秩（通常 r=8~64）

参数量对比：
原始：d × k
LoRA：r × (d + k)   ← 当 r 很小时，参数量大幅减少

例如：d=4096, k=4096, r=16
原始参数：16.7M
LoRA 参数：131K   ← 减少了 99.2%

为什么有效：

  • Aghajanyan 等人发现，大模型适应新任务时，参数更新的"内在维度"（intrinsic dimension）远小于参数空间维度。微调实际上只改变了少数"方向"。
  • LoRA 通过低秩分解显式地约束更新在这些低维子空间中进行。

LoRA 中的秩 r 和 α（scaling factor）分别控制什么？如何选择这些超参数？

高频调参实践
▶
核心回答
# LoRA 实际更新的公式
ΔW = (α / r) · B·A
#     ↑ scaling factor

其中：
r — 秩（Rank）：控制 LoRA 的表达能力
α — 缩放因子：控制 LoRA 更新的"强度"（类似学习率的角色）
超参数作用太小太大常用值
r低秩分解的秩 — 决定能捕获多少"变化维度"欠拟合，学不到足够知识过拟合 + 增加计算量8, 16, 32, 64
α缩放因子 — 放大/缩小 LoRA 更新的影响学得太慢训练不稳定通常 α = 2r（如 r=16, α=32）

选参指导：

  • 简单任务（单轮 QA、分类）：r=8, α=16
  • 中等复杂度（多轮对话、指令遵循）：r=16, α=32
  • 复杂任务（代码生成、长文推理）：r=32~64, α=64~128
  • 经验法则：α 设为 r 的 2 倍是一个好的起点
  • α 增大 ≠ 学习率增大。实际有效学习率 ≈ lr × (α/r)
8

LoRA 通常加在 Transformer 的哪些层？为什么主要加在 Attention 权重上？

实现细节
▶
核心回答

最常见的加 LoRA 的位置：

Transformer 层中可加 LoRA 的权重矩阵：

Multi-Head Attention:
  W_Q (Query)   ← ✅ 最常见
  W_K (Key)     ← ✅ 常见
  W_V (Value)   ← ✅ 最常见
  W_O (Output)  ← ✅ 常见

Feed Forward Network:
  W_up, W_gate, W_down  ← 可选，部分工作也加在这里

为什么侧重 Attention 权重：

  • Attention 权重是"信息路由"的关键——微调主要是改变模型"关注什么"和"如何组合信息"。
  • 原论文实验表明，只对 W_Q 和 W_V 加 LoRA 就能达到接近 Full FT 的效果。
  • 对 Attention 的 4 个权重矩阵全加（Q、K、V、O）通常效果最好。
  • 对 FFN 也加可以进一步提升，但会增加参数量和可能过拟合。

现代实践：大多数开源微调项目（如 LLaMA-Factory）默认对 Attention 的 Q、K、V、O 全部加 LoRA，即 target_modules=["q_proj","k_proj","v_proj","o_proj"]。

QLoRA 是如何在 LoRA 基础上进一步降低显存占用的？NF4 + 双重量化是什么？

高频必考前沿技术
▶
核心回答

QLoRA 在 LoRA 的基础上，通过量化主干模型（Base Model）进一步压缩显存：

显存占用对比（以 LLaMA-7B 为例）：

Full Fine-Tuning:  ~ 56 GB  (FP16 模型 + 梯度 + 优化器状态)
LoRA:              ~ 16 GB  (FP16 模型 + LoRA 梯度)
QLoRA:             ~ 10 GB  (NF4 量化模型 + LoRA 梯度)
                                           ↑ 单张消费级 GPU 即可

QLoRA 的三大技术：

  1. NF4（4-bit NormalFloat）

    一种针对正态分布权重优化的 4-bit 量化格式。假设预训练权重服从 N(0,σ)，NF4 在正态分布的尾部分配更多精度。比 INT4 更贴合神经网络权重的分布特点。

  2. 双重量化（Double Quantization）

    对量化过程中的缩放因子（scaling factor）再做一次量化（从 FP32 → FP8）。进一步节省 ~0.4 bits/参数。

  3. 分页优化器（Paged Optimizer）

    当显存不足时，将优化器状态换页到 CPU 内存（类似操作系统换页）。防止 OOM 而不影响训练正确性。

QLoRA 让在单张 24GB 消费级 GPU（RTX 3090/4090）上微调 33B 甚至 65B 模型成为可能。

如何合并（Merge）LoRA 权重到主模型？什么情况下应该 Merge，什么情况下不该 Merge？

工程实践
▶
核心回答
# LoRA Merge 公式
W_merged = W₀ + (α/r) · B·A

# 矩阵加法，将低秩适配器"融入"原始权重
# 合并后与普通模型完全一样，可以正常加载和推理

应该 Merge 的场景：

  • 生产部署：合并后不需要额外的 LoRA 加载逻辑，减少推理延迟。
  • 需要做量化部署：如 GPTQ / AWQ 量化通常需要完整的权重矩阵。
  • 模型共享/分发：合并后的模型更易用（用户不需要知道 LoRA）。
  • 多 LoRA 叠加：将多个 LoRA 逐个 merge 进同一个基底，实现能力组合。

不应该 Merge 的场景：

  • 需要服务多任务：保留基底 + 动态加载不同 LoRA 适配器，节省存储和显存。
  • 快速实验/迭代：LoRA 小文件（几 MB），可以快速切换和对比不同版本。
  • 隐私/安全隔离：不同用户使用不同的 LoRA，逻辑上隔离。

Adapter 和 LoRA 在架构上有什么本质区别？各有什么优劣？

对比分析
▶
核心回答
# Adapter 架构
x → [Self-Attention] → Add&Norm → [Adapter Layer] → Add&Norm → FFN → ...

Adapter Layer = W_up(ReLU(W_down(x))) + x
#  W_down: d → bottleneck → W_up: bottleneck → d
#  ↑ 串行插入，增加推理延迟

# LoRA 架构
x → [Self-Attention + LoRA] → Add&Norm → FFN → ...
#  ΔW = B·A 直接加到 W 上
#  ↑ 并行计算（可 Merge），不增加推理延迟
维度AdapterLoRA
位置串行插入在 Attention/FFN 之后并行加到 Attention/FFN 的权重上
推理开销有（多了层，必须串行计算）无（Merge 后等价于原始模型）
表达能力非线性（有 ReLU）线性（纯矩阵乘法）
参数量较多（通常 3-8%）较少（1-2%）
多任务需要切换 Adapter 层可动态热加载 LoRA 权重

什么是 Prefix Tuning 和 Prompt Tuning？它们与 LoRA 有什么本质不同？

方法理解
▶
核心回答

Prefix Tuning 和 Prompt Tuning 的核心思想：不修改模型权重，而是在输入空间中学习"软提示"。

# Prompt Tuning (Lester et al., 2021)
Input = [Soft_Prompt_1, ..., Soft_Prompt_k,  User_Input]
#        ↑ 可训练的虚拟 Embedding（k 个 token）
# 只在 Embedding 层加可训练参数，最简单但能力有限

# Prefix Tuning (Li & Liang, 2021)
# 在每一层都插入可训练的 Prefix 向量
for layer in transformer_layers:
    layer.attention.K = concat([Prefix_K, K])
    layer.attention.V = concat([Prefix_V, V])
# 每层都有 Prefix，表达能力更强

与 LoRA 的本质区别：

  • LoRA 修改的是权重空间（改变模型的计算方式）
  • Prefix/Prompt Tuning 修改的是输入/激活空间（改变模型的输入表示）
  • Prompt Tuning 的参数量是最少的（仅 Embedding 层）但效果也最弱

如何组合多种 PEFT 方法？例如 LoRA + Prefix Tuning 的场景和价值是什么？

进阶实践
▶
核心回答

不同 PEFT 方法作用在不同位置，可以正交地组合使用：

组合方案示例：

┌────────────────────────────────────────┐
│ LoRA (权重空间)                         │
│ → 学习领域知识、推理模式               │
│ → 加在 W_Q, W_V 上，r=32              │
├────────────────────────────────────────┤
│ Prompt Tuning (输入空间)                │
│ → 控制输出风格、安全行为               │
│ → 在 Embedding 层加 k=20 个软 token    │
├────────────────────────────────────────┤
│ IA³ (激活空间)                          │
│ → 快速调整任务偏好                     │
│ → 对 K, V, FFN 激活值缩放              │
└────────────────────────────────────────┘

实用场景：

  • LoRA（知识） + Prompt Tuning（风格）：LoRA 注入领域知识（如医学诊断），Prompt Tuning 微调输出风格（如"对儿童解释" vs "对医生解释"）。
  • LoRA 叠加多个 LoRA：将多个 LoRA 合并——如"医学 LoRA" + "友善语气 LoRA" → 友善的医学助手。

什么是 Instruction Tuning（指令微调）？它与传统 SFT 有什么不同？

高频必考核心概念
▶
核心回答

Instruction Tuning 是 SFT 的一种特殊形式，其训练数据格式为 (Instruction, Input, Output) 三元组，目标是让模型学会"遵循指令"而非简单"映射输入到输出"。

维度传统 SFTInstruction Tuning
数据格式任务特定的 (Input, Output)统一的 (Instruction, Input, Output)
目标优化特定任务的性能学会遵循各类指令
泛化性差（过拟合单一任务）强（跨任务泛化）
零样本能力无有（遵循未见过的指令）
代表单任务分类/翻译微调FLAN, Alpaca, Vicuna
# Instruction Tuning 数据格式
{
  "instruction": "将以下句子翻译成英文",
  "input": "今天天气真好",
  "output": "The weather is really nice today."
}

# 也支持无 input 的情况
{
  "instruction": "写一首关于春天的五言绝句",
  "input": "",
  "output": "春风吹绿柳，细雨润红花。燕子衔泥至，筑巢在檐下。"
}
💡 Instruction Tuning 是 ChatGPT 成功的核心要素之一：FLAN 论文表明，指令多样化的微调可以让模型获得零样本遵循新指令的能力。

如何构建高质量的 SFT 数据集？数据质量 vs 数据数量，哪个更重要？

高频数据工程
▶
核心回答

数据质量 > 数据数量。LIMA 论文（2023）表明：仅用 1000 条精心挑选的高质量 SFT 数据，就能让模型达到接近 GPT-4 级别的指令遵循能力。

高质量 SFT 数据的特征：

  1. 指令多样：覆盖不同任务类型（生成、推理、分类、创作、代码等），不同难度。
  2. 回答准确：事实正确、逻辑自洽、无幻觉。
  3. 格式统一：使用一致的 Chat Template（如 ChatML、Llama chat format）。
  4. 风格一致：语气、详细程度、输出格式保持一致。
  5. 安全无害：拒绝有害请求、不输出不安全内容。
  6. 多样性：覆盖不同领域（科技、人文、生活、法律等），不同语言。

数据构建 Pipeline：

1. 种子指令 → 2. 指令扩增 (Self-Instruct / Evol-Instruct) →
3. 强模型生成回答 (GPT-4 / Claude) → 4. 人工审核/修正 →
5. 质量过滤 (去重、去低质、安全检查) → 6. 格式统一化

Self-Instruct 和 Evol-Instruct 分别是什么？它们如何自动生成指令数据？

数据增强
▶
核心回答

Self-Instruct（自我指令生成）：

  1. 从一个种子池（175 条人工写的指令）开始
  2. 每次随机采样 8 条作为 Few-shot 示例
  3. 让 LLM 生成新的指令 → 判断是否为分类任务 → 生成回答 → 过滤低质量 → 加入池
  4. 迭代重复，最终从 175 条种子生成 82K 条指令（Alpaca 数据集）

Evol-Instruct（进化指令生成）：

  1. 对简单指令进行"进化"——让 LLM 将其变复杂
  2. 进化维度：加约束条件、深化推理、增加步骤、泛化场景、增加输入
  3. 再次让 LLM 生成新指令对应的回答
  4. 结果：WizardLM 数据集，包含大量复杂指令（多步推理、带约束生成等）
# Self-Instruct: 模仿 → 生成 → 过滤
种子 → 生成新指令 → 生成回答 → 质量过滤 → 扩展到 82K

# Evol-Instruct: 简单 → 进化 → 复杂
"写一首诗" → 进化 → "写一首七言律诗，主题是秋天，必须包含'枫叶'意象"

# 两者互补：Self-Instruct 负责广度，Evol-Instruct 负责深度

Chat Template 在 SFT 中起什么作用？不同模型的 Chat Template 有什么差异？

工程细节
▶
核心回答

Chat Template 定义了如何将对话结构化为 Token 序列——不同的 Base Model 使用不同的特殊 Token 和格式约定。

# Llama 3 Chat Template
<|begin_of_text|><|start_header_id|>system<|end_header_id|>
你是专业的Python编程助手。<|eot_id|>
<|start_header_id|>user<|end_header_id|>
请用Python实现快速排序。<|eot_id|>
<|start_header_id|>assistant<|end_header_id|>
以下是快速排序的Python实现：
def quicksort(arr):
    ...<|eot_id|>

# ChatML (Qwen / DeepSeek 等)
<|im_start|>system
你是专业的Python编程助手。<|im_end|>
<|im_start|>user
请用Python实现快速排序。<|im_end|>
<|im_start|>assistant
以下是快速排序的Python实现：
...<|im_end|>

为什么重要：

  • 用错 Template → 模型"看不懂"对话结构 → 性能大幅下降。
  • SFT 训练和推理必须使用完全一致的 Template。
  • 不同模型家族的特殊 Token 不同（Llama 用 <|eot_id|>，Qwen 用 <|im_end|>）。

HuggingFace 的 tokenizer.apply_chat_template() 可以自动应用每个模型对应的 Template。

SFT 数据中如何处理多轮对话（Multi-turn Conversation）？与单轮有什么不同？

数据处理
▶
核心回答
# 多轮对话数据格式
{
  "messages": [
    {"role": "user", "content": "Python中list和tuple的区别是什么？"},
    {"role": "assistant", "content": "list是可变的，tuple是不可变的..."},
    {"role": "user", "content": "那什么场景下应该用tuple？"},
    {"role": "assistant", "content": "当你需要不可变集合时，如作为dict的key..."},
    {"role": "user", "content": "能举一个具体的例子吗？"},
    {"role": "assistant", "content": "比如存储坐标点..."}
  ]
}

与单轮的关键区别：

  1. Loss 计算：每轮 assistant 回答都计算 Loss，所有 user 消息 Mask 掉。
  2. 上下文累积：每轮 assistant 的输出成为下一轮的输入上下文的一部分。
  3. 序列长度：多轮对话的总 Token 数可能很长，需要关注 max_length 截断策略（按轮截断 vs 按 Token 截断）。
  4. 连贯性：模型需要学习跨轮指代、话题跟随、追问应答等能力。

最佳实践：训练数据中混合单轮和多轮（如 30% 单轮 + 70% 多轮），让模型既能处理快速问答，也能胜任深度对话。

SFT 的关键训练超参数有哪些？如何设置学习率、Batch Size、Epoch 数？

高频调参指南
▶
核心回答
超参数常用范围设置建议
Learning Rate 1e-5 ~ 5e-4 LoRA 可用较高 LR（2e-4~5e-4）；Full FT 需低 LR（1e-5~5e-5）。太大→训练崩溃，太小→学不动。
LR Scheduler Cosine / Linear Cosine 最常用。通常配 3-10% warmup。
Batch Size 64 ~ 512 (global) 显存受限用 Gradient Accumulation 模拟大 Batch。micro_batch=4, grad_accum=16 → global_batch=64。
Epochs 1 ~ 5 SFT 通常 1-3 Epoch 足够。数据量小（<1万）可以用 3-5 Epoch，数据量大 1 Epoch 即可。多 Epoch 容易过拟合。
Max Sequence Length 2048 / 4096 / 8192 取决于任务+显存。长于 80% 的数据即可。太长浪费显存。
Warmup Ratio 0.03 ~ 0.1 前 3-10% 步 LR 从 0 线性增加到目标值，防止训练初期不稳定。
Weight Decay 0 ~ 0.1 LoRA 通常不需要（0），Full FT 用 0.01~0.1。
💡 经验之谈：LoRA 微调最关键的三个超参数 — Learning Rate（调得对不对看 Loss 曲线）、Rank r（太小欠拟合太大过拟合）、Epoch 数（多 Epoch 往往过拟合）。

什么是 Gradient Checkpointing 和 Gradient Accumulation？各解决什么问题？

显存优化
▶
核心回答

Gradient Checkpointing（梯度检查点）

  • 问题：反向传播需要保存前向传播的中间激活值，显存消耗巨大。
  • 方案：只保存部分层的激活值（检查点），反向传播时重新计算中间激活。用时间换空间。
  • 代价：训练速度降低约 20-30%。

Gradient Accumulation（梯度累积）

  • 问题：GPU 显存不够 → 无法用足够大的 Batch Size → 训练不稳定。
  • 方案：将一个大 Batch 拆分为多个 Micro-Batch，每个 Micro-Batch 前向+反向（但不更新），累积 N 个 Micro-Batch 的梯度后再做一次参数更新。
  • 代价：无（训练速度略降因为更多通信）。
# 示例：想用 global_batch_size=128 但单卡只能跑 4
per_device_batch = 4
gradient_accumulation_steps = 8
# 8 张卡 × 4 × 8 = 256 effective batch size

# 实现等价于：
for i, batch in enumerate(dataloader):
    loss = model(batch) / gradient_accumulation_steps
    loss.backward()  # 梯度累积
    if (i+1) % gradient_accumulation_steps == 0:
        optimizer.step()
        optimizer.zero_grad()

DeepSpeed ZeRO 的三个阶段分别做了什么？LLM 微调通常用哪个阶段？

高频分布式训练
▶
核心回答
阶段分片（Shard）的内容显存节省通信开销适用
ZeRO-1 Optimizer States ~4× 最低 优化器状态是瓶颈（大多数 Full FT 场景）
ZeRO-2 Optimizer States + Gradients ~8× 低 梯度也占显存的场景
ZeRO-3 Optimizer States + Gradients + Parameters 线性扩展（N 卡 ≈ N×） 较高 超大模型（100B+）训练

LLM 微调的推荐：

  • LoRA 微调：通常不需要 DeepSpeed，单卡 FSDP / DDP 就够。因为只有 LoRA 参数需要优化器状态。
  • Full FT (7B)：ZeRO-2 足够，4×A100(80G) 可以跑。
  • Full FT (70B+)：必须 ZeRO-3 + CPU Offload。
  • QLoRA：单卡 + Gradient Checkpointing 即可，接近不需 DeepSpeed。

如何判断 SFT 训练是否过拟合？有哪些监控指标？

训练监控
▶
核心回答

过拟合的信号：

  1. Train Loss 持续下降但 Eval Loss 上升：经典过拟合。Loss 分开走时可以停止训练。
  2. 生成重复/退化：模型开始输出重复短语、循环输出，或对所有问题都给出类似回答。
  3. Eval Loss 平坦：Eval Loss 不再下降，继续训练徒增过拟合风险。
  4. 多样性下降：对同一问题的不同问法，模型输出几乎相同。

监控体系：

  • 定量指标：Train/Eval Loss、Perplexity、Token Accuracy。
  • 定期生成检查：每 N 步用固定的一组 Prompt 生成回答，人工/LLM-as-Judge 检查质量。
  • Benchmark 评估：在 MMLU、HumanEval 等基准上评估通用能力是否下降（灾难性遗忘检测）。
  • KL Divergence：监控当前策略 vs 初始模型的 KL 散度（RLHF 中尤其重要）。
💡 核心原则：SFT 通常是 1-2 Epoch 就够了。不要追求 Training Loss 降到最低——Eval Loss 和实际生成质量才是你真正关心的指标。

LLM-as-Judge 是如何评估模型输出的？有什么优缺点？

评估方法
▶
核心回答

LLM-as-Judge 是让一个强 LLM（如 GPT-4、Claude Opus）作为裁判，给目标模型的输出打分或比较。

# 单模型打分
Judge Prompt:
"请从 1-10 分评估以下回答的质量，考虑准确性、完整性、流畅性和有用性。
问题：{question}
回答：{answer}
评分："

# 比较打分 (Pairwise)
Judge Prompt:
"以下是同一个问题的两个回答，请判断哪个更好。
问题：{question}
回答 A：{answer_a}
回答 B：{answer_b}
哪个更好？A 还是 B？为什么？"

优点：

  • 自动化，成本低于人工评估
  • 可扩展，能评估大量样本
  • 与人类偏好有一定相关性

缺点：

  • 位置偏差：倾向于选第一个/第二个（需做位置交换做两次取平均）
  • 长度偏差：倾向于给长回答高分
  • 自利偏差：GPT-4 可能偏好 GPT-4 风格的输出
  • 不一致性：同一对答案问两次可能得到不同结果
